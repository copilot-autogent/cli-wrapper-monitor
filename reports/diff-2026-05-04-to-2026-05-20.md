## Root Cause Analysis

### System Prompt Growth (+14.3%): Bootstrap file expansion

The system prompt assembles from four bootstrap files with a **20,000 char per-file limit**:

| File | May 4 (est.) | May 20 (actual) | Truncated? |
|------|-------------|----------------|------------|
| SOUL.md | ~4,500 chars | 4,517 chars | No |
| PLAYBOOK.md | ~12,000 chars | **39,673 chars** | **Yes** (first 20,000) |
| CONTEXT.md | ~17,000 chars | **41,739 chars** | **Yes** (first 20,000) |
| USER.md | ~8,500 chars | 11,872 chars | No |

**PLAYBOOK.md grew ~3.3×** in 16 days — driven by 10+ new sections added during active development (sprint ordering rules, bug tracking patterns, content placement guidelines, kickstart procedures, scheduled task checklist, investigation protocols, post-merge verification, side project handoff rules, and self-review guidelines).

**CONTEXT.md grew ~2.5×** — driven by accumulated gotchas, learnings, and operational notes added after incidents.

**Critical implication**: Both PLAYBOOK.md and CONTEXT.md now exceed the 20,000 char truncation limit. The model only sees the **first 20,000 chars** of each file. Later sections — including many recently added guidelines — are silently dropped from the context window. The agent is operating with incomplete operational guidelines.

## Interpretation

### Is this a problem?

**Context budget**: 16,126 tokens consumed at session start vs 12,956 (May 4). With claude-opus-4.6's 200k context window, this is 8.1% — still well within budget. The budget concern is secondary.

**Silent truncation is the real issue.** The system prompt truncation means:
1. New PLAYBOOK rules added in the second half of the file may not reach the model
2. New CONTEXT.md gotchas added recently are likely in the truncated tail
3. This creates an invisible gap between what the maintainer *thinks* the agent knows vs what it actually receives

**Tool growth is intentional but accelerating.** 18 tools added in 16 days is fast. The playwright suite (9 tools) and data workbench (3 tools) were coherent feature additions. At this rate, tool definitions will consume ~10,000 chars/month.

---

## Recommended Actions

1. **Raise `maxCharsPerFile` config** — from 20,000 to 30,000 or 40,000 chars to accommodate the current file sizes. Files are now 2× the per-file limit.

2. **Audit PLAYBOOK truncation point** — identify which sections fall after the 20,000 char mark in PLAYBOOK.md and CONTEXT.md. These are silently invisible to the model.

3. **Consider file splitting** — PLAYBOOK.md at 39,673 chars is a monolith. A `PLAYBOOK-CORE.md` (always loaded) + `PLAYBOOK-EXT.md` (loaded on demand) architecture would be more efficient.

4. **Add `bootstrapTruncated` metric to baseline schema** — this sprint introduced it; it should be a permanent fixture.
